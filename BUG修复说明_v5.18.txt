一、修复的BUG
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【BUG #38】meta文件字段名错误 (严重性: 致命)
  
  问题描述:
  - 中转机脚本在生成meta配置文件时，错误地使用了 TRANSIT_IP 字段名
  - 实际存储的是落地机IP，但字段名却是 TRANSIT_IP
  - 导致map文件中写入了错误的IP地址（中转机IP而非落地机IP）
  
  影响:
  - 所有节点完全不通
  - 流量被转发到中转机自己，形成死循环
  - v5.17及之前的所有版本都受影响
  
  修复:
  - 将 meta 文件中的字段名从 TRANSIT_IP 改为 LANDING_IP
  - 修复位置: 第1525行 _atomic_apply_route() 函数
  - 修复代码:
    printf 'DOMAIN=%s\nLANDING_IP=%s\nPORT=%s\nUUID=%s\nPWD=%s\nPFX=%s\nCREATED=%s\n'

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【方案A：手动修复现有配置（推荐，快速）】

前提条件：你需要知道每台落地机的真实公网IP地址

步骤1：在每台落地机上获取IP
  curl -4 ifconfig.me

步骤2：在中转机上修复配置文件
  假设你有3台落地机：
  - cf.998488.xyz 对应落地机IP: 1.2.3.4
  - gg01.998488.xyz 对应落地机IP: 5.6.7.8
  - gg02.998488.xyz 对应落地机IP: 9.10.11.12

  执行以下命令：

  # 修复 cf.998488.xyz 的map文件
  echo "cf.998488.xyz    1.2.3.4:8443;" > /etc/nginx/stream-snippets/landing_cf_998488_xyz_*.map

  # 修复 gg01.998488.xyz 的map文件
  echo "    gg01.998488.xyz    5.6.7.8:8443;" > /etc/nginx/stream-snippets/landing_gg01_998488_xyz_*.map

  # 修复 gg02.998488.xyz 的map文件
  echo "    gg02.998488.xyz    9.10.11.12:8443;" > /etc/nginx/stream-snippets/landing_gg02_998488_xyz_*.map

  # 测试配置
  nginx -t

  # 重载Nginx
  systemctl reload nginx

步骤3：验证修复
  # 从中转机测试到落地机的连接
  nc -zv 1.2.3.4 8443
  nc -zv 5.6.7.8 8443
  nc -zv 9.10.11.12 8443

  # 应该都显示 "succeeded" 或 "open"

步骤4：测试客户端连接
  使用客户端连接中转机IP的443端口，应该可以正常使用

四、技术细节
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

问题根源：
  在 import_token() 函数中，从token解析出的 'ip' 字段是落地机的IP地址。
  但在 _atomic_apply_route() 函数中，这个IP被错误地标记为 TRANSIT_IP。
  
  错误的代码（v5.17）：
    printf 'DOMAIN=%s\nTRANSIT_IP=%s\nPORT=%s\n' "\" "\" "\"
  
  正确的代码（v5.18）：
    printf 'DOMAIN=%s\nLANDING_IP=%s\nPORT=%s\n' "\" "\" "\"

影响链：
  1. token中的ip字段 = 落地机IP
  2. import_token解析后存入变量 \
  3. _atomic_apply_route将\（错误！）
  4. map文件中也使用了\，但因为字段名错误，导致逻辑混乱
  5. 最终结果：中转机不知道要转发到哪里

修复后的正确流程：
  1. token中的ip字段 = 落地机IP
  2. import_token解析后存入变量 \
  3. _atomic_apply_route将\（正确！）
  4. map文件使用\
  5. 最终结果：中转机正确转发流量到落地机

六、常见问题
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q1: 我已经安装了v5.17，必须重新安装吗？
A1: 不需要。使用方案A手动修复配置文件即可，只需要几分钟。

Q2: 如何知道落地机的IP地址？
A2: 在落地机上执行 curl -4 ifconfig.me 即可获取。

Q3: 修复后还是不通怎么办？
A3: 请检查：
    - 落地机防火墙是否放行了中转机IP
    - 落地机的8443端口是否正常监听
    - 从中转机能否ping通落地机
    - 云服务商安全组是否开放了相应端口

Q4: 我有多台落地机，需要全部修复吗？
A4: 是的，每台落地机对应的map文件都需要修复。
